Quantization Basics

Building a shared mental model before introducing AutoRound

What Is Quantization?

Quantization is data mapping — it maps a large set of continuous floating-point values onto a small set of discrete integer levels, using fewer bits per weight/activation.

[Image: ./images/quant-element.png]

Mapping continuous FP values onto a discrete integer grid — scale sets the step, rounding picks the level, the leftover gap is the error.

① quantize
xint = round( xfloat / scale )
② de-quantize
xdequant = xint × scale
③ cost   error = xfloat − xdequant

① Scale — step size of the grid
scale = max(|x|) / (2n−1−1). Larger → wider range, coarser steps; smaller → finer steps but clips outliers.

② Rounding — snap to nearest level
Default is round-to-nearest (RTN). AutoRound's insight: up vs. down can be learned — nearest isn't always best.

③ Quantization error — what mapping costs
Gap between original and recovered value (MSE / accuracy drop). Good quantization minimizes it.

0.234 → round(0.234/0.172)=1 → 0.172 → err 0.062

Why Quantization Is Necessary for LLM/VLM Serving

The Hardware Reality

① Model Footprint

  • Runnable vs. not runnable — weights too large → shard (adds cost) or can't serve
  • Free memory = KV cache headroom → larger batches, longer context, higher throughput

② Computation Efficiency

Compute-bound = GPU waits on math units. Memory-bound = GPU waits on data from HBM.

  • Compute-bound — e.g. prefill. Tensor Cores: 2–4× at low bits (needs activation quant)
  • Memory-bound — e.g. decode (small batch size). Bandwidth is fixed, less data → lower latency. Qualifier: at high concurrency, decode shifts toward compute-bound.

AutoRound: How It Works & Where It's Used

Algorithm, engineering, community adoption — and why it's practical for deployment

The Core Problem: Why Low-Bit Quantization Is Hard

It's hard not because errors exist, but because they fight each other — you can't minimize all three at once, and a fixed bit budget gives you nowhere to hide.

① Outlier Amplification

One big outlier sets the scale, so small weights collapse toward 0.

scale = max(|x|)/(2n−1−1)
[0.1, 0.2, 8.0] → scale set by 8.0
0.1, 0.2 (low-bit) → both round to 0

Why hard: LLM weights/activations are full of outliers you can't drop — one bad value silently destroys many others.

② Rounding Error

A value lands between two grid levels and snaps to the nearest one.

round(0.47)→0.4
error = 0.07

Why hard: fixed bit budget — fewer bits force bigger steps. You can't make the grid finer without spending bits you don't have.

③ Clipping Error

Shrink the range for finer steps, and large values fall outside → clamped.

clip(0.95, qmax=0.6)→0.6
error = 0.35

Why hard: no free lunch — widen the range to stop clipping and rounding gets coarser. Fixing one error worsens the other.

They share one knob — the clip threshold (= scale). Rounding error and clipping error are the same trade-off seen from two ends: tighten the range → less rounding error but more clipping; widen it → the reverse. Outliers just push the U apart, raising the best error you can reach.

And it compounds across layers, while the rounding choices form a 2N combinatorial space RTN can't search.

AutoRound Algorithm: The Big Picture

Papers: arxiv.org/abs/2512.04746 · EMNLP 2024

Quantization is a discrete optimization problem — we can use gradient-based optimization if we make the rounding decision differentiable.

What AutoRound Optimizes

ParameterShapeControlsInitial
v (rounding)same as WRound up vs. down0 (nearest)
min, max (clipping)scalar / groupQuantization rangemin(W), max(W)
 
W_quant = clamp(round(W/scale + v), min_int, max_int)
W_dequant = W_quant * scale

v is bounded to [−0.5, 0.5] — it controls whether each weight rounds up or down at the quantization boundary.

Why It Works

  • Block-wise optimization — captures actual downstream impact
  • SignSGD — robust to noisy STE gradients
  • Joint optimization — rounding + clipping together

Algorithm Flow

▼
For each transformer block:
▼

▼

▼

▼
④ Update: SignSGD
v ← v − lr × sign(grad) · Update clipping similarly
▼

▼
Output Quantized + packed checkpoint

SignSGD: The Optimization Engine

Why SignSGD?

Standard SGD: w = w − lr × gradient
SignSGD:       w = w − lr × sign(gradient)
  • Fixed step size naturally handles the discrete problem
  • We find SignSGD sufficient and stable for the rounding sub-problem

SignRound V1 → V2

V1 (EMNLP 2024): SignSGD on rounding and clipping

V2 (arXiv 2025):

  • ① Adaptive mixed-precision — leverages gradient information and quantization-induced reconstruction errors to guide layer-wise bit allocation
  • ② Lightweight stabilization — loss filtering and pre-tuning scale search to improve tuning effectiveness in extremely low-bit regimes (2-bit, MXFP4)

SignRound V1 (EMNLP 2024)
aclanthology.org/2024.findings-emnlp.662

SignRound V2 (arXiv 2025)
arxiv.org/abs/2512.04746

AutoRound uses SignSGD because the sign of the gradient is more reliable than its magnitude when optimizing through a non-differentiable quantization operation.

Quantize FLUX

⚠️ Check Your Device First

  • What data types does it natively support?
  • What's the Memory Capacity?
  • What's the TFLOPS (prefill bottleneck)?
  • Match the scheme to what your hardware actually accelerates

No mature kernel → no speedup, regardless of bit-width.
