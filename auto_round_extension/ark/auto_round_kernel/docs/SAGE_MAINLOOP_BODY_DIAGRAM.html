SAGE mainloop_body(...) Diagram

This note explains what one call to mainloop_body does inside wrapper/include/stla/xe_sagev1_fwd_mainloop.hpp. The key point is that one workgroup already owns a fixed Q tile and output V tile; mainloop_body processes one selected K tile for that workgroup.

Workgroup owns one Q tile
Workgroup owns one output V tile
One call handles one K tile
Dense and sparse differ only in K traversal order

In this file, the words tile and block are often used for the same kernel-level unit. For the Wan D=128 path, one QK step is a score tile of 256 query rows × 32 key rows, computed through a head-dimension K-slice of size 64.

3. How This Maps to Code

Step 1. Resolve which K tile to load. Dense mode passes contiguous K values. Sparse mode passes only selected K blocks from lut_row. Cached mode first maps to cache/current physical tile indices through k_idx.
Step 2. Compute one QK score tile. The body copies one Q fragment and one K fragment, reorders them for subgroup MMA, then runs cute::gemm(mma_qk, tSrQ, tSrK, tSrS).
Step 3. Transform scores into probabilities. The score tile is scaled with qscale/kscale, then causal/full/tail masks are applied, then online softmax updates the row max/sum state.
Step 4. Multiply by the matching V tile. The body prefetches V, then runs P × V for each output V subtile and accumulates into tArA. This is repeated over many K tiles. In the Wan D=128 launch family, one PV subtile covers 64 output channels, so two subtiles fill the full 128-wide head output.

4. Dense vs Sparse Traversal

Dense path

for K = blk_k0 .. blk_k1-1:
  mainloop_body(K)

The workgroup visits every K tile in a contiguous range. This is why dense K prefetch can use K + Stages.

Sparse path

cur_block = 0
for i in selected_blocks:
  cur_block += lut_row[i]
  mainloop_body(cur_block)

The workgroup only visits LUT-selected K tiles. That sequence may be non-contiguous, so the dense prefetch rule is not valid here.

5. What “block” Means Here

In this file, “block” means a tile of the attention matrices, not a full attention head and not a whole tensor.

  • Q block: a tile of query rows by full head dimension
  • K block: a tile of key rows by full head dimension
  • S block: the score tile generated from one Q block and one K block
  • V block: the matching value rows used by that same K tile
Wan D=128 translation:
Q block = 256 query tokens × 128 head channels
K block = 32 key tokens × 128 head channels
S block = 256 query tokens × 32 key tokens
V block per PV step = 32 key tokens × 64 output channels
One call to mainloop_body does not finish the full attention result. It updates the output accumulator with the contribution from one selected K block. Repeating it over all dense or sparse K blocks finishes the output tile.
