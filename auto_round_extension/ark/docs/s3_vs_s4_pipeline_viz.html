S4 vs S3 WOQ GEMV — the compute pipeline (ARK XPU, Battlemage)

Both assign one sub-group (32 lanes) per output column and split the K dimension across lanes. Same shape ⇒ same useful work. Yet S4 runs at ~453 GB/s (DRAM peak) and S3 at ~137 GB/s. The difference is entirely what each lane does per step — visualized below.

The calibration that proves it's compute, not memory: at n=k=4096, S2 (2-bit, reads 4.2 MB) takes 0.041 ms while S4 (4-bit, reads more data: 8.4 MB) takes 0.0185 ms. The kernel that moves the least data is slower → none of the sub-4-bit kernels are bandwidth-bound. They stall on per-weight decode/accumulate overhead before reaching the memory ceiling. S4 is the only one that drove overhead low enough to saturate DRAM.

S4 — WeightS4T::gemv 453 GB/s · peak

TileK0=8, Unroll=2, PrefetchDis=2. Each lane owns 8 contiguous K-weights per inner step.

One lane's slice = 8 weights packed in 4 bytes (2 nibbles/byte). 32 lanes × 8 = 256 K-weights per unrolled slice, read as one coalesced 128-byte transaction.

S3 — WeightS3T::gemv (SLM-coalesced) 137 GB/s · ~3× off

SgSize=32, Unroll=2. Each lane owns a 32-weight group = 3 straddled words per step.

One lane's slice = 32 weights packed in 3 words (12 bytes), fields cross word boundaries. 4× more weights per lane per step than S4 → heavier, lumpier iterations.

Takeaway: S3's deficit is the red/orange column — SLM+barriers and the branchy 32-wide straddle decode — not its 3-bit size. Porting S4's barrier-free, branchless, tightly-tiled compute structure onto the existing dense 3-bit blob keeps 3-bit memory while removing the overhead that caps it at 137 GB/s.
