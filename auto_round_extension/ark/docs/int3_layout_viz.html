int3 WOQ packing layouts & the scale-block alignment problem

ARK XPU GEMV — why the dense 3-bit blob is GPU-load-hostile, and how "10 values per 32-bit word" fixes it. Block size = 128 (one scale per 128 K-elements).

Legend: value in scale-block A value in scale-block B wasted / padding bits word that needs 2 scales (bad) word needs 1 scale (good)

1  Today: dense 3-bit straddle layout

32 values are packed into exactly 3 words (96 bits = 32×3, zero waste). But a 3-bit field can cross a word boundary: value #10 occupies bits 30–32, i.e. 2 bits in word 0 and 1 bit in word 1.

↑ Each cell = 1 bit. Notice values #10 and #21 are split across two words (the straddle). Decoding them needs an if (crosses boundary) branch and reads from two words.

GPU problem — uncoalesced load. The natural mapping is "lane L decodes 32-value group L", so lane L must read words 3L, 3L+1, 3L+2. Across 32 lanes the first-word addresses are 0, 12, 24, 36 … bytes — a stride-12 gather, not a contiguous block. The hardware can't fuse it into one transaction → ~86 GB/s vs S4's ~480 GB/s.
lane0→bytes[0..11]  lane1→bytes[12..23]  lane2→bytes[24..35]   ✗ stride-12, scattered

2  Proposed: 10 values per 32-bit word (drop top 2 bits)

Pack 10 × 3-bit = 30 bits into one word; waste the top 2 bits. Now no value ever crosses a word boundary → decode is a pure (w >> 3*i) & 0x7, no branch. And lane L reads word L → 32 lanes read 32 contiguous words = one coalesced 128-byte load, exactly like the fast S4 path.

↑ 10 clean 3-bit slots + 2 wasted bits per word. Cost: 3.2 bits/value (+6.7%) instead of 3.0.

lane0→word0[bytes 0..3]  lane1→word1[bytes 4..7]  lane2→word2[bytes 8..11]   ✓ contiguous, coalesced

3  The catch: scale-block alignment (128 ÷ 10 = 12.8)

A new scale starts every 128 values. But each word now holds 10 values, and 128 is not a multiple of 10. So the word that contains value #128 holds values 120–129 — values 120–127 belong to block A, values 128–129 belong to block B. That one word needs two different scales.

Words 0–11 (values 0–119): all block A. Word 12 (values 120–129): mixed ✗. Words 13+: block B…

↑ value-level view around the block boundary. The red-bordered word #12 straddles the A/B scale boundary — the bit-straddle is gone, but a scale-straddle took its place.

Scheme A — pad each block to whole words recommended

Round each 128-block up to ceil(128/10)=13 words (130 slots; pad the last 2). Every word now belongs to exactly one block → scale = sptr[word / 13]. Fully word-aligned, trivial kernel.

13 words/block, last word = 8 valid + 2 pad. 3.25 bits/val (+8.3%).

Scheme B — dense, split the boundary word

Keep the packed stream fully dense (no per-block padding). Only the ~1 boundary-crossing word per block decodes its two halves with two scale indices.

~1 mixed word per 128 block. 3.2 bits/val (+6.7%), slightly more decode logic.

Why 10-per-word wins: it gets the coalesced, branchless access of the 4-bit repack while keeping almost all of int3's memory advantage (+6.7% vs +33%). The only new complexity is the scale-block alignment handled above. Speedup is expected to land near the S4 reference but must be measured.
