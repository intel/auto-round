Is the gap that sycl-tla can't do group-wise scale?

Short answer: no. It does group-wise scale fine. The gap is one level deeper, and it's structural — about where the u4→s8 conversion runs, not whether the scale can be expressed. Here it is in pictures.

Fix the mental model first

✗ the wrong model

"sycl-tla's mixed collective can't apply a group-wise weight scale, so it can't match ARK."

This is what the confusion comes from — and it's false.

✓ what's actually true

sycl-tla's mixed collective has a full group-wise path. I ran it at --g=128 and it passed verification.

Evidence: is_groupwise (xe_mma_mixed_input.hpp:175), runtime group_size (:273), scale_k = ceil_div(K, g) (:320), and the --g knob in the example (:123).

Three capabilities — and nobody but ARK has all three

Group-wise scale is just one column. Notice sycl-tla ticks it in the mixed row. The column it can't tick in that same row is the rightmost one.

The heart of it: where does the u4→s8 conversion live?

Same conversion, same arithmetic. The only difference is position. In sycl-tla it rides inside the hot loop and re-runs every k-tile. In ARK it's lifted into a one-time DRAM pass, leaving the hot loop pure.

sycl-tla — mixed mainloop

one kernel · conversion welded in
// the ONLY group-wise-capable mainloop
for blk in groups:           // group-wise ✓
  for ktile in blk:        HOT LOOP · ×K/32
    load B_u4[ktile]      // narrow
    CONVERT u4 → s8       // ✗ EVERY ktile
    load A[ktile]
    MMA  acc_s32 += A·B   // DPAS
  acc ×= scale[blk]       // group-wise ✓ (cheap)

ARK — two-pass

conversion hoisted · hot loop stays pure
// PASS 1 — unpackq kernel
B_u4 ─CONVERT u4→s8─▶ B_s8 (DRAM) ONCE
// PASS 2 — IKblockGemmDQCore
for blk in groups:           // group-wise ✓
  for ktile in blk:        HOT LOOP · pure
    load B_s8[ktile]      // already wide ✓
    load A_s8[ktile]
    MMA  acc_s32 += A·B   // PURE DPAS
  acc_f32 += acc_s32 × scale[blk] // ✓
u4→s8 conversion (the cost) pure int8 DPAS (the fast part) group-wise scale (cheap, both have it)
Look at the orange in each diagram. It's the same operation in both. On the left it sits inside the hot loop → it runs K/32 times and never amortizes. On the right it's been pulled out into Pass 1 → paid once, then the hot loop is wall-to-wall green. That hoist is the entire difference. Group-wise scale (blue) is identical on both sides.

Proof it isn't the scale — and isn't even the conversion math

If the bottleneck were the scale, both fused variants would be slow. If it were the cost of the conversion arithmetic, the cheaper conversion would be faster. Neither holds. Watch what happens when we only change the conversion target:

sycl-tla fused variantin-loop conversionrelative cost of that convertm512 TFLOP/s
FP16U4FP16S8 u4 → s8 cheap (bit unpack) 3.5
FP16U4FP16F16 u4 → fp16 expensive (int→float) 14.2
The path with the cheaper conversion is 4× slower. So the cost isn't the scale, and it isn't the conversion's arithmetic either — it's structural: the mixed mainloop's int8 staging is untuned / poorly pipelined on BMG, and the per-k-tile reload never amortizes. Even the faster fused number (14.2) still loses 4.6× to ARK's 65.1 — because fp16 DPAS is only ½ the int8 peak.

So, precisely, the gap is…

the one-liner

NOT "sycl-tla can't do group-wise scale."
IT'S "sycl-tla has no performant mainloop that pairs group-wise-scaled int32 accumulation with a pure s8-DPAS inner loop."

It has group-wise scale. It has fast s8 DPAS. It just never has both in one tuned loop — its only group-wise path welds the u4→s8 conversion into the hot loop. ARK gets all three by decoupling: widen once into DRAM (unpackq), then run a tight group-wise-scaled s8 loop (IKblockGemmDQCore). That hoist — not group-wise support — is the thing sycl-tla is missing.
