ARK XPU int3 (S3) WOQ GEMV — Optimization Journey

Battlemage / Intel Arc Pro B60 Graphics  ·  SYCL, m=1 weight-only-quant GEMV  ·  branch feat/ark-xpu-int3-woq-gemm  ·  all variants bit-exact (max_diff = 0)

TL;DR

  • Goal: close the ~2.5× gap between int3 (S3) GEMV and the int4 (S4) reference kernel.
  • Best result: a transposed (lane-interleaved) weight layout — +16% / +15% over the plain nested kernel (0.899 / 0.843 TFLOPS). Still ~2× short of S4.
  • Decisive finding (low-level profiling): int3 GEMV achieves only ~180 GB/s = 25% of the 725 GB/s DRAM ceiling. It is latency/occupancy-bound, NOT bandwidth-bound — it moves fewer bytes than S4 yet runs 2.5× slower.
  • Two premises were disproven mid-investigation by reading the GPU ISA (see callouts). Every "obvious" lever was tested and ISA-verified rather than assumed.

3. Step-by-step

Step 0 — Nested baseline (0.774 / 0.733)

Each lane owns whole blocks; the inner word loop is unrolled via integer_sequence so every bit offset, valid-count, and activation base is a compile-time constant — no div/mod, no straddle branch, scale loaded once per block. This alone was the prior big win (+45–95% over the dynamic-index version). It plateaued at ~150 GB/s. The question: why?

Step 1 — ILP sweep FLAT

Hypothesis: the single serial bacc += a*v chain caps IPC at ~1/latency; S4 uses 2 accumulators. Split per-block FMAs across NACC=2/4/8 independent accumulators.

Result: dead flat (<1%). ISA confirmed FMA count unchanged (128/127/121 mad) and no register spills through nacc=8. Conclusion: the plateau is NOT an FMA-latency stall. ILP ruled out as the bottleneck.

ISA correction #1 — the plateau is a memory-feed stall, not compute.
Dumping the nested kernel's load instructions showed ~40 per-lane gathers (load.ugm.d32x4.a64 (32|M0)) per step vs S4's far fewer. The (32|M0) + per-lane address register = a SIMD32 scatter/gather. The lane-owns-whole-blocks scheme reads block L strided by WPB across the sub-group → non-contiguous → gather. This redirected all subsequent work toward the load pattern.

Step 2 — Hand-rolled SLM staging REGRESSION

Idea: stage block words into SLM with slm[p*32+sg_id]=base[p*32+sg_id] (consecutive lanes → consecutive words) to coalesce.

Result: slower (0.700 / 0.625). ISA verdict: the staging still compiled to 86 gathers (up from 40) + 21 SLM ops + 13 barriers, zero block loads. Paid the SLM+barrier tax without earning coalescing. Manual scalar SLM copies don't lower to a block load.

Step 3 — Padded vector load +5% / +1%

Idea: pad each block's 13 words to 16 (pow2) so each lane issues one vec<uint32_t,16> load — mirroring S4's per-lane vector-load structure.

Result: small real win, but goal not met. ISA: weight load stayed load.ugm.d32x4.a64 (32|M0) gather (37, was 40). The vector only widened the per-lane message (d32x1→d32x4); it did not coalesce across lanes. Root cause: lane L reads block L at 64-byte stride → across-lane span still non-contiguous.

Step 4 — Transposed lane-interleaved layout BEST: +16% / +15%

The structural fix. Group blocks into tiles of 32; within a tile store words word-major, lane-minor: word (wb, lane L) lives at tilebase + wb*32 + L. Now when all 32 lanes read their block's word wb via tilebase + wb*32 + sg_id, the 32 addresses are contiguous.

plain nested:   lane L reads block L  -> addresses stride by WPB  -> gather scattered across DRAM
transposed:     lane L, word wb       -> tilebase + wb*32 + L      -> 32 contiguous words per wb

Result: best variant, 0.899 / 0.843 TFLOPS, exact. (k=4096 = exactly one 32-block tile → zero padding → its 182.7 GB/s is honest.)

ISA correction #2 — the win is real but NOT via a block-load instruction.
The transposed kernel still emits load.ugm.d32x4.a64 (32|M0) gathers (46), not S4's d32xNt transpose loads. The speedup comes from a different effect: contiguous per-word lane addresses let the hardware coalesce the gather into fewer cache-line transactions at runtime. Coalesced traffic, gather instruction. The instruction form was never the lever.
(Caught mid-analysis: IGC entry-numbers reshuffle every rebuild — re-mapped S4/S3 by mangled name to avoid misreporting.)

Step 5 — Software prefetch FLAT

Motivated by ISA correction #3 (below). Add joint_prefetch of whole tiles PrefetchDis=3 ahead, on top of the transposed layout.

Result: flat (0.899 vs 0.890 — noise). ISA verified the prefetch DID emit (8× load.ugm.d32x4t.a64.uc.ca (32|M0)... null:0 — S4's exact null-dest, cache-hinted prefetch form), so this is a true negative, not a missed intent.

ISA correction #3 — I had misread S4 the whole time.
S4's d32x16t/d32x4t "block loads" I'd been chasing for three steps have null:0 destinations = they write no register. They are software prefetches (emitted by S4's 4 joint_prefetch calls), NOT its weight loads. S4 demand-loads its weights with the same d32x4.a64 (32|M0) gather family as our int3 kernels. The "block-load vs gather" gap I'd hunted for three variants never existed.

4. The bandwidth verdict

With prefetch flat, I measured what actually limits the kernel — achieved DRAM bandwidth vs the bus ceiling.

PathAchieved read BW% of peakReading
DRAM ceiling (pure-read probe)725 GB/s100%the bus
S4 GEMV (weight blob)~450 GB/s62%healthy occupancy
int3 transposed (best)~180 GB/s25%leaving 3/4 on the floor
This refutes the "int3 is bandwidth-bound" hypothesis.
int3 moves fewer bytes than S4 (0.81× — the 3-bit blob is genuinely smaller) yet runs 2.5× slower, at only 25% of the 725 GB/s bus. If it were bus-bound, less data would mean faster. It is latency / occupancy-bound: not enough memory requests in flight to saturate the bus. This also explains why prefetch was flat — the bus isn't the constraint; request concurrency is.

5. Where the residual ~2× lives — and what to try next

Every instruction-level difference between int3 and S4 was tested and ISA-verified. The remaining gap is occupancy / load-parallelism: one sub-group per output column with a load-then-immediately-consume structure issues ~1 outstanding load per lane — too few to hide DRAM latency. Ranked next levers:

  1. Multiple output columns per sub-group — issue NCols weight reads, then decode/FMA → NCols× independent loads in flight, no extra barriers. Most direct attack on the 25% ceiling.
  2. Deeper register prefetch — load several tiles into registers ahead of the consuming FMAs to decouple issue from consume.
  3. More sub-groups per work-group (4–8) — more resident threads to hide latency. Pure occupancy knob.
  4. Denser pack (drop W10's 2 wasted bits/word) — ~6% fewer bytes; orthogonal, small, only stack on (1).

Do NOT retry (ISA-disproven this session): more accumulators alone, SLM staging, chasing the block-load instruction, software prefetch.

6. Method note

The throughline: every hypothesis was checked against the generated ISA or a direct measurement, not assumed. Three times the ISA overturned a plausible premise (compute-bound → memory-feed; block-load gap that didn't exist; S4's "block loads" were prefetches). Two "obvious" wins (ILP, prefetch) were flat and one (SLM) regressed — only the layout change, targeting the proven mechanism, actually moved the needle. All variants kept in the harness (test_gemm.hpp, namespace w10_spike) as reproducible evidence.
