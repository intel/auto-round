▼ UMD stack — downloaded as .deb from GitHub releases
IGC
Intel Graphics Compiler (IGC) — SPIR-V → GPU ISA JIT
The heart of the shader compilation chain. Takes generic SPIR-V from the SYCL device compiler and lowers it to native GPU instructions (Xe2 ISA on Battlemage). Ships as two debs from the IGC GitHub releases.
github.com/intel/intel-graphics-compilerrelease v2.24.8
intel-igc-core-22.24.8+20344core compiler: spir-v → vISA → GPU binary
intel-igc-opencl-22.24.8+20344OpenCL built-ins + SPIR-V extensions
IGC is embedded inside the compute-runtime — it runs as a library (libigc.so), not a standalone binary. Every time the L0 driver loads a SPIR-V module, IGC JIT-compiles it. This is why driver version = IGC capability = which SPIR-V ops work on a given node.
▼ compute-runtime UMD: L0 driver + OpenCL + ocloc + GMM

🔗 Runtime Data Flow — how a vLLM XPU forward pass traverses this stack

1. vLLM Python calls torch.xpu matmul / attention op
2. PyTorch dispatches to IPEX or oneDNN (libdnnl.so) → SYCL queue
3. libsycl.so selects the Level-Zero backend, hands SPIR-V kernel + args to the L0 loader
4. L0 loader routes to libze-intel-gpu.so (compute-runtime UMD)
5. UMD calls IGC (libigc.so) to JIT-compile SPIR-V → Xe2 ISA
6. UMD builds a command buffer, submits it via DRM ioctl to the kernel driver
7. xe/i915 KMD schedules on GPU → GuC firmware loads context → Xe-cores execute
8. (multi-node) KV cache transfer goes through NIXL → UCX → libze-intel-gpu GPU-direct RDMA
9. (multi-GPU) allreduce / allgather goes through oneccl_bind_pt → oneCCL → L0

⚠ Version Compatibility Notes

oneAPI 2025.3 lock
DPC++ compiler, MKL, and DNNL must stay on 2025.3. The base image (2025.3.2) and apt pin (version 2025.3*) enforce this. Bumping the base image requires updating both the pin and the UMD to match.
oneCCL decoupled from oneAPI
The default oneAPI 2025.3 CCL does not support Battlemage. The 2021.15.9.14 release from uxlfoundation does — so it is installed separately and the pip-bundled oneccl is removed.
UMD version lock
IGC 2.24.8 + compute-runtime 25.48.36300.8 + L0 1.26.0 are a matched set. The IGC and ocloc versions must be compatible with the libze-intel-gpu build. Mismatched debs cause zeInit failures or SPIR-V lowering errors.
Triton XPU replaces triton
Upstream triton does not target Intel GPUs. triton-xpu==3.7.0 is the Intel fork. Both must be uninstalled before installing the pinned version to avoid import path conflicts.
How to read this: each colored block is a distinct package group from docker/Dockerfile.xpu. The stack builds bottom-up: base image → oneAPI apt → UMD debs → oneCCL → pip wheels → vLLM build → NIXL/UCX → triton.

Key insight: the UMD stack (neo/*.deb) is the most version-sensitive layer. IGC version determines which SPIR-V ops lower correctly to GPU ISA; the compute-runtime version determines which IGC is embedded. These three repos (IGC, compute-runtime, level-zero) must be bumped together when upgrading.
